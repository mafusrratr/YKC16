# Zhongshihua2.0 已完成帧类型清单

本文档用于记录 `core/comm/Zhongshihua2.0` 当前已落地的协议帧，实现状态以代码为准（`comm_process.h/.cpp`）。

## 1. 登录与在线流程（已实现）

- `0x01` 充电桩登录认证（上行，已实现）
  - 已组帧发送，包含登录信息体。
- `0x02` 登录认证应答（下行，已实现）
  - 已解析登录结果；
  - 失败回到 `LOGIN_IDLE`；
  - 成功进入 `LOGIN_REQ_FEE_MODEL`。
- `0x09` 计费模型请求（上行，已实现）
  - 当前按“仅桩编号”请求。
- `0x0A` 计费模型请求应答（下行，已实现）
  - 已解析并保存计费模型；
  - 按整桩生效更新全部枪；
  - 收到有效 `0x0A` 后触发发送 `0x0B`，再进入在线态。
- `0x0B` 对时请求（上行，已实现）
  - 信息体为桩编号。
- `0x0C` 对时请求应答（下行，已实现）
  - 已解析 `CP56Time2a` 并执行系统对时。
- `0x03` 心跳包（上行，已实现）
  - 登录完成后按周期发送（当前20秒）。
- `0x04` 心跳包应答（下行，已实现）
  - 已接收处理（在线维持）。

## 2. 业务控制帧（已实现）

- `0x14` 远程启动（下行，已实现）
  - 已解析平台下发启动参数；
  - 已转发 MQTT：`tcu/plat/{gun}/cmd`，`cmd=start_charge`；
  - 计费模型参数可同步保存用于后续上报。
- `0x15` 远程停机（下行，已实现）
  - 已解析并转发 MQTT：`tcu/plat/{gun}/cmd`，`cmd=stop_charge`；
  - 停机原因按当前约定固定 `0x01` 下发。

## 3. 运行与结果上报（已实现）

- `0x13` 充电信息上报（上行，已实现）
  - 在枪状态 `CHARGING/STOPPING` 时按周期（10秒）上报；
  - 数据来源已接入：
    - `tcu/pile/{gun}/data`（yx/yc）
    - `tcu/meter/{gun}/data`
    - `tcu/logic/{gun}/feeData`
- `0x2D` 启动充电结果上报（上行，已实现）
  - 触发源：`tcu/pile/{gun}/event` 的 `start_complete`；
  - 判定规则：`successFlag==0` 为成功，否则失败；
  - 失败原因取 `start_complete.chargeFailReason`。
- `0x19` 上传充电结束阶段报文（上行，已实现）
  - 触发源：`tcu/pile/{gun}/event` 的 `stop_complete`；
  - `stopSoc/单体最低最高电压/电池最低最高温度` 取 `stop_complete`；
  - `本次累计充电时间/本次充电电量` 取 `tcu/logic/{gun}/feeData` 缓存。
- `0x60` 充电记录包请求（上行，已实现）
  - 触发源：`tcu/logic/{gun}/event` 的 `update_record`；
  - 已完成组帧发送。
- `0x70` 充电记录包请求应答（下行，已实现）
  - 已解析确认结果；
  - `feedbackResult` 为 `0x00` 或 `0x04` 视为成功；
  - 成功后调用 `m_logSender.confirmTradeRecord(...)` 确认本地记录。

## 4. 帧加密处理（当前状态）

- 已具备“按帧类型判定是否加密”的机制；
- 登录相关密钥准备、会话密钥与公钥更新链路已接入状态机；
- 具体 SM2/SM4 强安全实现按当前工程依赖逐步完善（现阶段先保障协议流程联调可跑通）。

## 5. 当前登录状态机

- `LOGIN_IDLE`
- `LOGIN_REQ_AUTH`（发送 `0x01`，等待 `0x02`）
- `LOGIN_REQ_FEE_MODEL`（发送 `0x09`，等待 `0x0A`）
- `LOGIN_ONLINE`（发送 `0x03` 心跳；处理业务收发）
