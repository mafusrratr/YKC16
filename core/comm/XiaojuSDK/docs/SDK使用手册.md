一、 简介
1.概述
  SDK 是小桔为充电桩开发者提供的一套软件包，目的是为了用户更快速的接入小桔充电平台；目前主要实现了 MQTT 协议、小桔充电桩与服务器通信协议。
  SDK 内部实现了数据周期上报、电量计算、账单计算、协议解析、用户回调等功能，提供了一些桩端所需的操作接口，用户只需拷贝特定文件夹的文件即可实现移植。
  SDK 针对常用嵌入式系统，实现了对应 demo 实例供用户参考。

二、程序开发
1.启动充电
1.简述
  SDK 内部已经实现了对充电相关协议的解析，桩端只需要实现启动充电的回调接口。
  启动充电时序图:
参与方：用户 / 小桔平台 / 充电桩。
1. 用户执行插枪或拔枪动作。
2. 充电桩向小桔平台发送插枪事件。
3. 用户扫码充电。
4. 小桔平台向充电桩下发 `code=7`。
5. 桩端实现充电回调，SDK 回复 `code=8`。
6. 桩端调用 `touch_send_bms_basic_info` 发送 BMS 基础信息。
7. 桩端调用 `xj_send_touch_charge_start`，通知平台启动充电。
2.实现步骤
 i.实现 callback_start_charge 充电回调函数。
 说明:
    如果无法启动充电,需要返回非"0000"的故障码,此时不会产生 202 账单,后台判定桩充电启动失败。
    如果可以进入充电状态，需返回"0000",此时 SDK 内部进入自检状态，账单数据开始计算。后台判定桩处于自检状态。需要注意的是,SDK 需要等该回调结束后才会发送 code=8，但如果超过 90s 未传 code=8，后台会认为启动超时。
    为防止上述问题出现异常，平台建议桩端可以在 callback_start_charge 中直接返回"0000"，自检启动过程在其他地方处理
 ii. 调用 touch_send_bms_basic_info 发送 bms 基础数据
 说明:
    桩端自检完成后，无论是否可以启动充电都需要发送一次 bms 基础数据
 iii. 调用 xj_send_touch_charge_start 发送启动充电事件
    由于桩端自检过程可能过长,造成后台因超时判定充电失败，但桩端却进入充电流程，造成状态和数据不同步情况。现通过 code=8 和 code=108 共同判断是否进入充电状态。
    自检结束后如启动失败，xj_send_touch_charge_start 的 param 参数需要写入失败原因(故障码),sdk 会发送 0 元账单；如启动成功 param 写入"0000"
3.实例
  1. void callback_start_charge(uint8_t gun_cnt,char* err,char* id,uint8_t id_len,uin
      t16_t user_tel)
  2. {
  3.      /* do something */
  4. /* 30s 内返回 禁止阻塞*/
  5.      strcpy(err,XJ_SUCCEED_RESULT);//模拟正常情况
  6. }
  7. /* 发送 bms 基础数据 */
  8. touch_send_bms_basic_info(...);
  9. /* 发送 108 充电事件 通知后台桩启动成功 */
  10. xj_send_touch_charge_start(gun_index,XJ_SUCCEED_RESULT);
2022-01-19 变更 增加两个参数
charge_policy(充电模式)
0:充满为止（默认）
1:时间控制充电
2:金额控制充电
3:电量控制充电
4:按 SOC 控制充电
Charge_policy_param(模式对应的参数)
时间单位为 1 秒
金额单位为 0.01 元
电量时单位为 0.01kw
SOC 单位为%
  11. void callback_start_charge(uint8_t gun_cnt,char* err,char* id,uint8_t id_len,uint16_t user_tel,uint8_t charge_policy,uint32_t charge_policy_param)

2.停止充电
停止充电分为 APP 停止、平台限充停止、桩正常停止、桩异常停止
APP 停止充电时序图:
参与方：用户 / 小桔平台 / 充电桩。
1. 用户在 APP 点击停止。
2. 后台向充电桩发送停止充电命令 `code=11`。
3. 桩端实现停止充电回调，SDK 回复 `code=12`。
4. 桩端调用 `touch_send_bms_inish_statistical_data` 发送 BMS 统计数据。
5. 桩端调用 `xj_send_touch_charge_stoped` 通知后台停止充电。
6. SDK 发送 202 账单。
1.实现步骤
  i. 实现 callback_stop_charge 停止充电回调
    桩端可以在该回调中做一些停止充电的工作。但是 sdk 还未上传 202 账单。后台也未判定桩端以停止充电。
  ii. 调用 touch_send_bms_inish_statistical_data 发送 bms 统计数据
 说明：
     桩端必须发送一次 bms 统计数据
  iii. 调用 xj_send_touch_charge_stoped 发送停止充电事件
 说明：
    桩端在最终停止充电时，需要调用该函数通知后台桩已经停止充电。此时sdk 会上传 202 账单
2.实例
  1. void callback_stop_charge(uint8_t gun_cnt,char* err)
  2. {
  3.      /* do something */
  4.      strcpy(err,XJ_SUCCEED_RESULT);//模拟正常情况 仅代表收到停止命令
  5. }
  6.
  7. touch_send_bms_inish_statistical_data();
  8. xj_send_touch_charge_stoped(gun_index,"1001",XJ_SUCCEED_RESULT);
  9. /* sdk 自动上报 202 账单 */
  10. xj_send_touch_gun_pluged_out();

平台限充停止
平台限充停止时序图:
参与方：小桔平台 / 充电桩。
1. 桩端周期发送 104 数据。
2. 后台判断 `SOC > 99`，下发停止充电指令 `code=11`。
3. 桩端需要实现停止回调，并发送 `code=12`。
4. 桩端调用 `touch_send_bms_inish_statistical_data` 发送 BMS 统计数据。
5. 桩端调用 `xj_send_touch_charge_stoped` 通知后台停止充电。
6. SDK 发送 202 账单。
1.实现步骤
和 APP 停止的区别是后台会对 104 报文的 SOC 值做判断，超过 99 时自动下发停止充电指令
2.实例

  2. {
  3.      /* do something */
  4.      strcpy(err,XJ_SUCCEED_RESULT);//模拟正常情况 仅代表收到停止命令
  5. }
  6.
  7. touch_send_bms_inish_statistical_data();
  8. xj_send_touch_charge_stoped(gun_index,"1001",XJ_SUCCEED_RESULT);
  9. /* sdk 自动上报 202 账单 */
  10. xj_send_touch_gun_pluged_out();

桩正常停止
1.实现步骤
桩端屏幕点击停止充电或判断 SOC 充满时调用以下函数停止充电。
touch_send_bms_inish_statistical_data();
xj_send_touch_charge_stoped(gun_index,"xxxx",XJ_SUCCEED_RESULT);
2.实例
  1.   touch_send_bms_inish_statistical_data();
  2.   xj_send_touch_charge_stoped(gun_index,"1001",XJ_SUCCEED_RESULT);
  3.   /* sdk 自动上报 202 账单 */
  4.   xj_send_touch_gun_pluged_out();

桩异常停止
桩异常停止时序图:
参与方：用户 / 小桔平台 / 充电桩。
1. 用户触发急停或发生强制拔枪。
2. 桩端上报 118 故障（或 120 告警）。
3. 桩端上报 BMS 统计信息。
4. 桩端发送 108 停止充电事件。
5. 若为强制拔枪，桩端发送 108 拔枪事件。
6. SDK 发送 202 账单。
1. 实现步骤
   i. 调用 xj_send_touch_error_occured 发送具体故障
 说明：
  后台对 118 故障的判断优先级最高，上报故障未恢复时，后台还是判定桩处于故障状态。对于不影响下次充电的故障可不通过 118 上报。
   ii. 调用 touch_send_bms_inish_statistical_data 发送 bms 统计数据
   iii. 调用 xj_send_touch_charge_stoped 发送停止充电事件
 说明：
  注意在调用 xj_send_touch_charge_stoped 前还需要发送一次 bms 统计数据，此时 SDK 会自动上报 202 账单
   iiii. 调用 xj_send_touch_gun_pluged_out 发送拔枪事件(拔枪停充时发送)
 说明:
  如果在充电中强制拔枪(理论上不允许该操作),可先发送告警事件。再发拔枪事件。如果以 118 发送，后台会一直认为故障。但是拔枪不影响下次充电。
  如果充电中强制拔枪，必须先调用 xj_send_touch_charge_stoped, 上报停充事件，这时 SDK 会自动上报 202 账单，再触发 xj_send_touch_gun_pluged_out,拔枪事件 CODE 108, 否则会导致 202 账单上报未知异常停止
2.实例
  1. /* 平台下发停止需要实现*/
  2. void callback_stop_charge(uint8_t gun_cnt,char* err)
  3. {
  4.      /* do something */
  5.      strcpy(err,XJ_SUCCEED_RESULT);//模拟正常情况 仅代表收到停止命令
  6. }
  7. /* 情况 1.强制拔枪停止 */
  8. touch_send_bms_inish_statistical_data();
  9. xj_send_touch_charge_stoped(gun_index,"100d",XJ_SUCCEED_RESULT);
  10. xj_send_touch_warning_occured(gun_index,"100d");
  11. xj_send_touch_gun_pluged_out(gun_index);
  12.
  13. /* 情况 2.急停等内部故障导致停充 */
  14. xj_send_touch_charge_stoped(gun_index,err_code,XJ_SUCCEED_RESULT);
  15. xj_send_touch_error_occured(gun_index,err_code);

3.故障上报
1.简述
SDK 提供了上报故障和恢复故障的操作接口。桩端只需调用即可。
充电中故障上报流程图:
参与方：用户 / 小桔平台 / 充电桩。
1. 用户按下急停，或桩端发生其它故障。
2. 桩端上报 `code=118` 故障。
3. 平台回复 117（必须回复）。
4. 充电情况下，桩端调用 `touch_send_bms_inish_statistical_data` 发送 BMS 统计数据。
5. 充电情况下，桩端调用 `xj_send_touch_charge_stoped` 停止充电。
6. 充电情况下，SDK 发送账单。
  当故障发生后需要按照实际的故障码调用 xj_send_touch_error_occurred 函数，故障恢复后按照恢复的故障码调用xj_send_touch_error_recovered
  重新上电后，平台会根据 106 中的“启动次数”字段值变化来清空所有故障，该字段值 SDK 已做上电递增处理
  注意在充电中产生故障时，除了调用 xj_send_touch_error_occurred 发送故障，还需要发送一次 bms 统计数据，然后调用 xj_send_touch_charge_stoped 通知平台已经停止充电

4.告警上报
当充电桩需要上报告警但不影响充电时,可调用 xj_send_touch_warning_occured发送告警事件；对于存在阈值的告警可调用 xj_send_warning 函数发送

5.二维码设置
SDK 已实现二维码字符串生成算法和平台下发特定二维码功能。用户仅需根据提供的字符串展示二维码即可，即实现 callback_display_qr 函数。SDK 会在启动时和桩编号被成功修改时调用

6.校时设置
充电桩连接平台后需要主动向平台发送签到指令(106)，不签到后续指令会被丢弃。目前 SDK 中已经加入了签到指令的实现逻辑。
当桩发送 106 签到报文时，后台回复的 105 报文中带有系统时间。桩端需要实现callback_set_sys_time 回调，将时间写入 RTC 中

7.在线升级
在线升级时序图:
参与方：用户 / 小桔平台 / 充电桩。
1. 用户/平台下发升级指令。
2. 后台向充电桩发送 `code=1101`。
3. 桩端根据指令执行升级操作。
4. 桩端回复 `code=1102`。
5. 升级完成后，桩端发送 106，后台确认软件版本变更。
目前后台仅支持 HTTP 方式升级。用户可在运维平台手动下发升级指令。
平台下发升级时只把下载的 url 地址发送给充电桩(code=1101)，桩端需要实现 callback_software_download 回调函数。通过 http 的 get 请求来实现在线升级。

8.日志上传
日志上传时序图:
参与方：用户 / 小桔平台 / 充电桩。
1. 用户/平台拉取本地日志。
2. 后台向充电桩发送 `code=409`。
3. 桩端上报 `code=410`。
4. 桩端通过 HTTP POST 请求传输日志。
SDK 中是实现了 410 和 409 协议。
桩端需要实现 callback_upload_log 回调函数，通过 http 的 post 请求来上传日志。协议头中的 content-type 字段设置为 multipart/form-data；http 消息体中需要配置一个 deviceid=<桩号>的参数
SDK 中实现了日志打印接口 xj_pal_print_log.但需要桩端实现 User_log_export函数(xiaoju_pal.c)来实现日志的输出路径

9.账单展示
当账单结束后。SDK 内部会将当前账单作为参数传入 callback_show_bill 函数中。桩端可以直接展示到屏幕。或者将该账单保存到本地。
在充电过程中，SDK 内部将当前账单作为参数传入 callback_save_bill 函数中。该回调作用是数据固化到 flash 或 eeprom 中。桩端可以不实现。但是异常情况有可能出现账单数据错误情况(例如整桩断电)。
在系统启动时，SDK 会调用 callback_read_bill 回调函数。将未上传或未收到 201 回复的账单重新上报。

10.数据固化
SDK 中的 xj_params 结构体保存的是需要配置的可变参数。SDK 启动时会去读本地
配置文件。需要桩端实现 xj_pal_read_persist_params(数据读取)和
xj_pal_write_persist_params(数据写入)两个函数(xiaoju_pal.c)

11.VIN 码启动充电
1. 简述
SDK 即插即充流程图：
1. 用户插枪，桩端触发 `xj_send_touch_gun_pluged_in` 插枪事件，事件码 `CODE 108`。
2. 界面触发“即插即充按钮”。
3. 桩端启动车辆交互：`CHM-BHM` 交互、`CRM-BRM` 交互。
4. 判断是否获取到有效 VIN 码：
   - 否：界面提示即插即充启动失败；无需调用 SDK 接口与平台交互。
   - 是：进入 VIN 码在线鉴权流程。
5. VIN 码在线鉴权流程：
   - 调用 SDK 接口 `xj_send_vin_start_charge_request` 发起 VIN 码在线鉴权。
   - 27930 流程按正常交互到准备就绪阶段。
   - 持续发送 `CRO=0x00` 报文，直到接收 VIN 码鉴权结果返回（超时时间 10 秒）；鉴权结果返回且预充完成后，发送 `CRO=0xAA`。
6. 平台进行 VIN 码在线鉴权，并通过 `callback_vin_start_charge_result` 返回鉴权结果；结果包含 VIN 码鉴权结果和预充完成状态。
7. 结果分支：
   - 鉴权成功且预充完成：进入充电界面；调用 `touch_send_bms_basic_info` 上报 BMS 基础信息 `CODE 304`；调用 `xj_send_touch_charge_start` 上报启动成功，`param` 传入 `"0000"`，事件码 `CODE 108`。
   - 鉴权失败或鉴权超时：界面提示即插即充启动失败原因；无需调用 SDK 接口与平台交互。
   - 充电流程交互失败但 VIN 码鉴权成功：界面提示启动失败；执行停止充电流程；调用 `touch_send_bms_basic_info` 上报 BMS 基础信息 `CODE 304`；调用 `xj_send_touch_charge_start` 上报启动失败，`param` 传入非 `"0000"` 的失败原因，事件码 `CODE 108`；随后上传 202 账单。

VIN 码启动充电主要步骤时序图:
参与方：用户 / 小桔平台 / 充电桩。
1. 用户在屏幕点击 VIN 码启动。
2. 桩端发送 VIN 码鉴权请求。
3. 后台回复鉴权结果。
4. 桩端调用 `callback_start_charge` 启动充电。
2.主要步骤
SDK 提供了 VIN 码启动充电的接口 xj_send_touch_vin_charge_reques，并将鉴权结果以参数形式传入 callback_vin_start_charge_result 函数。桩端根据实际需求启动充电。

12.费率展示
为了便于桩端展示费率信息，SDK 提供了三个接口用于获取 48 个时段的电价、服务费、延迟费。接口如下。
  1.   /*获取 0~48 个任意时段的电价*/
  2.   uint16_t xj_get_charge_fee(uint8_t time);
  3.
  4.   /*获取 0~48 个任意时段的服务费*/
  5.   uint16_t xj_get_service_fee(uint8_t time);
  6.
  7.   /*获取 0~48 个任意时段的延迟费*/
  8.   Uint16_t xj_get_demurrage(uint8_t time);


13.安全防护工作参数
为了使充电更加安全，SDK 提供了防护参数。桩端需要根据回调接口下发的参数实
现安全防护的功能。

函数原型:
void callback_safety_protect_param(xj_l2_safety_prot_param*param,uint8_t *err)
此函数是以单次充电为单位，不需要做存储处理，触发此函数时则进行防护操作，否则不进行任何操作。

14.并充上报
当充电桩选择并充或解除并充时，充电桩需上报此时并充的枪号以及并充标志，桩端只需调用即可。注意：调用此函数需启动充电前。只支持同桩并充。
函数原型:
int8_t xj_send_co_changing(uint8_t major_gun,uint8_t minor_gun,uint8_t charge_type)
当并充发生时调用 xj_send_co_changing 函数，参数 charge_type 为真(即为 1)
当并充接触时调用 xj_send_co_changing 函数，参数 charge_type 为假(即为 0)

杭州计量改造的设备结束充电时，上报加密数据。
int8_t xj_send_meter_encryp_data(uint8_t gun_cnt,char* id,char *meter_addr, char *encry_data)

16.功率控制
平台下发功率控制时，调用此函数。
gun_cnt :枪号(0-MAX) kwh:功率
int8_t callback_power_control(uint8_t gun_cnt,uint32_t kwh)

三、如何移植
主要步骤：
1.拷贝 src 和 include 文件夹中的内容到自己的工程目录。
2.根据所用系统环境拷贝 porting 目录下对应的 xiaoju_pal.c 和 xiaoju_pal.h 文
件到自己的工程目录。
3.实现 xiaoju_pal.h 和 xiaoju_pal.c 中的函数接口和回调函数。
说明：
SDK 的 porting 目录可以理解为一个操作系统模拟层。它提供了一个公共接口,用于屏蔽操作系统和网络模块的实现细节。用户需要根据实际系统环境实现里面的任务创建销毁、互斥锁、时间操作以及各个回调函数。
所有接入平台的桩企必须实现联网功能，因联网方式多种多样，SDK 中仅提供了网络读写接口，实际接入方式需用户自行实现。
SDK 提供的对外接口存放在 include 目录下,逻辑实现在 xiaoju.c 中。
4.填充 st_user_bms_info 和 st_user_gun_info 结构体内容
说明：
SDK 将枪状态信息和 bms 信息以透传方式传递给平台。对于某些字段的精度和偏移需要用户自行处理后传入 SDK。
使用步骤：
1. 修改 xiaoju_pal.h 中的逻辑服务地地址(XJ_LOGIC_SVR_ADDR)、端口号(XJ_LOGIC_SVR_PORT)、桩号(XJ_EQUIPMENT_ID)、用户名(XJ_MQTT_USERNAME)、密码(XJ_MQTT_PASSWORD)。
2. 填充用户端枪信息和 BMS 信息结构体；调用 xj_APP_start(st_user_gun_info* gun_status, st_user_bms_info* bms_status,uint8_t gun_num)，执行 SDK 逻辑。
3. 成功连接平台后，调用 xiaoju.h 中的接口执行响应操作

四、注意事项
1.参数配置
目前与平台的连接方式为直连逻辑服务器，用户在联调测试完毕后，需要将逻辑服务器地址改为正式服务器地址，使用宏“XJ_LOGIC_SVR_ADDR”和“XJ_LOGIC_SVR_PORT”注释后面的地址，即 epower-equipment-server.xiaojukeji.com 1884。
“XJ_MQTT_USER_NAME”和“XJ_MQTT_PASSWORD”为 MQTT 登录时的用户名和密码，SDK 提供的为测试账号，正式上线需要向相关人员索要并修改。
“USER_VERSION”为用户程序版本号，CODE106 签到时会体现，与远程升级模块相关，用户更新代码建议修改此版本号。
2.数据格式
所有接口中与枪号有关的参数，均为 0 表示枪 1(A 枪) ,1 表示枪 2(B 枪)，以此类推。
平台发往充电桩的数据为 ISO 格式的，充电桩需要使用 utf-8 格式，所以收到的数据需要进行一次转码。SDK 已经完成了 ISO 到 utf-8 的协议转码，用户获取到数据是转码后的
3.接入域名
目前小桔平台都是使用域名方式连接，有的用户前期调试时 DNS 可能有问题，解析不出 IP 地址。调试期间可以使用我们提供的 IP 接入，后面需要改成域名方式。
正式逻辑服务器：
epower-equipment-server.xiaojukeji.com vip 116.85.2.49:1884
测试逻辑服务器：
epower-equipment-server-test.xiaojukeji.com vip 116.85.2.54:1884

七、接入流程
1.SDK 移植
用户移植 SDK，实现充电逻辑。程序开发期间遇到问题均可邮件或其他方式沟通
2.测试联调
用户开发完成后。需要和小桔开发人员共同联调测试。完成数据完整性、充电主流程和一些异常情况的校验测试。
联调期间小桔会分配测试桩号。用户可在运维平台或接入平台上查看桩传入数据和实时状态。
用户扫码测试需要下载"滴滴车主"APP,使用小桔充电功能测试扫码充电启动。
3.场站公测
联调测试结束后，小桔人员会分配正式桩号和用户密码。在公测阶段我们会持续监测公测桩的异常率，如出现某些车型充电不成功等现象，请协助我们共同查找问题
