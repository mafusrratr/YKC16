# 消息队列大小配置指南

## 当前配置

代码中尝试设置消息队列大小为 **1MB**（优先选择，应对爆发式错误日志），如果系统不允许则依次降级到 **512KB**、**256KB**。

```cpp
// 尝试设置为 1MB（优先选择）
buf.msg_qbytes = 1024 * 1024;
// 如果失败，尝试 512KB
buf.msg_qbytes = 512 * 1024;
// 如果仍然失败，尝试 256KB
buf.msg_qbytes = 256 * 1024;
```

## 容量计算

### 消息大小分析

| 消息类型 | 典型大小 | 说明 |
|---------|---------|------|
| 普通日志 | 150-300 字节 | JSON格式，包含时间戳、模块、消息等 |
| 充电记录 | 500-800 字节 | 包含多个时段数据 |
| 计费模型 | 1000-2000 字节 | 96时段模型可能达到 2000+ 字节 |
| 最大单条消息 | 8192 字节 | 系统限制（msgmax） |

### 容量估算

假设平均消息大小为 **200-300 字节**：

- **256KB**: 约 850-1300 条消息
- **512KB**: 约 1700-2600 条消息
- **1MB**（当前配置）: 约 3400-5200 条消息
- **2MB**: 约 6800-10400 条消息

## 512KB 是否够用？

### 场景分析

#### 场景1: 正常负载（推荐场景）
- 日志频率：10-50 条/秒
- 处理速度：100-500 条/秒（logger进程）
- **结论**: 512KB 完全够用，队列基本不会堆积

#### 场景2: 高负载（压力测试）
- 日志频率：500 条/秒
- 处理速度：100-200 条/秒（数据库写入是瓶颈）
- 队列增长：约 300 条/秒
- **1MB 容量**（当前配置）: 约 4000 条消息
- **填满时间**: 4000 ÷ 300 ≈ **13.3 秒**
- **结论**: 提供更好的缓冲能力，可以承受更长时间的突发流量

#### 场景3: 突发流量
- 短时间内大量日志（如错误爆发）
- 如果处理速度跟不上，512KB 可能在几秒内填满
- **结论**: 需要更大容量作为缓冲

### 建议

**当前配置 1MB 对于大多数场景是足够的**，包括：

1. **高并发系统**: 多个进程同时发送日志
2. **突发流量**: 短时间内大量日志（如错误爆发）
3. **数据库写入慢**: logger 处理速度低于发送速度
4. **爆发式错误日志**: 系统异常时可能产生大量错误日志

如果仍然遇到队列填满的问题，可以考虑：
- 进一步增大到 2MB（仅在极端场景）
- 优化处理速度（数据库批量写入、提高处理频率）

## 设置更大的影响

### 优点

1. **更大的缓冲能力**
   - 可以承受更长时间的突发流量
   - 减少消息丢失的风险
   - 给 logger 进程更多处理时间

2. **更好的容错性**
   - 即使处理速度暂时下降，也不会立即填满
   - 系统有更多时间恢复

### 缺点和限制

1. **系统资源占用**
   ```
   消息队列内存 = msg_qbytes × 消息队列数量
   
   例如：
   - 4个队列 × 512KB = 2MB
   - 4个队列 × 2MB = 8MB
   ```
   - 对于嵌入式系统，内存是有限资源
   - 但通常几MB的内存占用是可以接受的

2. **系统限制**
   ```bash
   # 查看系统限制
   cat /proc/sys/kernel/msgmax   # 单条消息最大大小（通常 8192）
   cat /proc/sys/kernel/msgmnb   # 消息队列总容量默认值（通常 16384 = 16KB）
   cat /proc/sys/kernel/msgmni   # 系统最大队列数（通常 32000）
   ```
   - `msgmnb` 是默认值，可以通过 `IPC_SET` 设置更大的值
   - 但某些系统可能有硬性限制（需要 root 权限修改）

3. **延迟增加**
   - 队列越大，消息在队列中等待的时间可能越长
   - 但对于日志系统，这通常是可以接受的（异步处理）

4. **恢复时间**
   - 如果队列填满，需要更长时间才能清空
   - 但这也意味着系统有更多缓冲时间

## 推荐配置

### 嵌入式系统（内存有限）

```cpp
// 当前配置：1MB（优先选择，应对爆发式错误日志）
buf.msg_qbytes = 1024 * 1024; // 1MB
// 如果系统不允许，自动降级到 512KB 或 256KB
```

**理由**:
- 平衡内存占用和缓冲能力
- 1MB 对于大多数场景足够，包括突发流量
- 内存占用可接受（1MB 对于现代嵌入式系统不算大）
- 可以更好地应对爆发式错误日志记录

### 高负载系统

```cpp
// 推荐：1MB - 2MB
buf.msg_qbytes = 1024 * 1024;  // 1MB
// 或
buf.msg_qbytes = 2 * 1024 * 1024; // 2MB
```

**理由**:
- 可以承受更高的突发流量
- 给系统更多恢复时间
- 2MB 内存占用仍然可接受

### 极端场景

```cpp
// 推荐：2MB - 4MB（仅在必要时）
buf.msg_qbytes = 2 * 1024 * 1024;  // 2MB
// 或
buf.msg_qbytes = 4 * 1024 * 1024;   // 4MB
```

**理由**:
- 仅用于极端高负载场景
- 需要确保系统有足够内存
- 通常不推荐超过 4MB

## 动态调整策略

当前代码已经实现了智能降级：

```cpp
// 尝试 512KB
if (设置失败) {
    // 降级到 256KB
    if (仍然失败) {
        // 使用系统默认值（通常 16KB）
    }
}
```

### 建议改进

可以添加配置选项，允许用户自定义：

```cpp
// 从配置文件读取
int queueSizeKB = config.getInt("MessageQueue", "size_kb", 512);
buf.msg_qbytes = queueSizeKB * 1024;
```

- **持续满队列**: 需要增大容量或提高处理速度

## 总结

| 场景 | 推荐大小 | 理由 |
|------|---------|------|
| 正常负载 | **1MB**（当前配置） | 足够，可以应对突发流量 |
| 高负载 | **1MB**（当前配置） | 更好的缓冲能力，应对爆发式错误日志 |
| 极端场景 | 2MB | 仅在必要时使用 |
| 内存受限 | 256KB | 最小可用配置（自动降级） |

**当前 1MB 配置是合理的**，对于大多数场景足够使用，包括爆发式错误日志记录。如果遇到队列经常填满的问题，建议：

1. 先尝试优化处理速度（数据库写入优化）
2. 如果仍然不够，再考虑增大到 2MB（仅在极端场景）
3. 监控队列使用率，根据实际情况调整
