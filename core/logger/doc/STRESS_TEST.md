## 概述

本文档说明 Logger 模块的压力测试功能，用于验证日志系统在高负载情况下的性能和稳定性。

压力测试程序 `test_logger` 包含基本功能测试和压力测试两部分，可以全面测试日志系统的各项功能。

## 测试内容

### 基本功能测试

在进入压力测试之前，程序会先执行以下基本功能测试：

1. **常规日志测试** - 发送不同级别的日志消息（INFO/WARN/ERROR）
2. **TradeRecord 测试** - 测试合法和非法充电交易记录的校验
3. **FeeModel 测试** - 测试合法和非法计费模型的校验
4. **共享内存测试** - 验证计费模型是否正确同步到共享内存

### 压力测试

压力测试包含 5 个测试场景，全面测试系统性能：

#### 测试 1: 大量日志消息快速发送

**测试内容：**
- 快速发送 10,000 条日志消息
- 测试消息队列和数据库写入性能

**测试参数：**
- 消息数量：10,000 条
- 消息类型：INFO 级别日志
- 进度显示：每 1,000 条显示一次

**预期结果：**
- 显示发送总耗时（毫秒）
- 显示吞吐量（消息/秒）
- 所有消息成功发送到消息队列

#### 测试 2: 多线程并发发送日志

**测试内容：**
- 5 个线程并发发送日志
- 测试并发安全性和消息队列处理能力

**测试参数：**
- 线程数量：5 个
- 每线程消息数：1,000 条
- 总消息数：5,000 条

**预期结果：**
- 显示并发发送总耗时
- 显示并发吞吐量（消息/秒）
- 验证多线程环境下消息不丢失、不重复

#### 测试 3: 大量 TradeRecord 快速发送

**测试内容：**
- 快速发送 1,000 条充电交易记录
- 测试大数据量 TradeRecord 处理性能

**测试参数：**
- 记录数量：1,000 条
- 每条记录包含：完整充电信息、时段电量、费用等
- 进度显示：每 200 条显示一次

**预期结果：**
- 显示发送总耗时
- 显示吞吐量（记录/秒）
- 所有记录成功入库

#### 测试 4: 大量 FeeModel 快速发送

**测试内容：**
- 快速发送 100 个计费模型（每个包含 96 时段）
- 测试最大数据量处理性能

**测试参数：**
- 模型数量：100 个
- 每模型时段数：96 个（最大时段数）
- 进度显示：每 20 个显示一次

**预期结果：**
- 显示发送总耗时
- 显示吞吐量（模型/秒）
- 验证大数据量计费模型处理能力

#### 测试 5: 混合负载测试

**测试内容：**
- 同时发送日志、TradeRecord 和 FeeModel
- 模拟实际使用场景的混合负载

**测试参数：**
- 总消息数：500 条
- 日志消息：500 条
- TradeRecord：每 10 条消息发送 1 个（共 50 个）
- FeeModel：每 50 条消息发送 1 个（共 10 个）
- 进度显示：每 100 条显示一次

**预期结果：**
- 显示混合负载处理总耗时
- 显示混合吞吐量（消息/秒）
- 验证系统在混合负载下的稳定性

## 使用方法

### 1. 启动 Logger 进程

在运行压力测试之前，必须先启动 Logger 进程：

```bash
# 启动 logger 进程（后台运行）
cd /usr/app/tcu
./tcu_logger /usr/app/config/logger.ini &
```

### 2. 运行测试程序

```bash
# 运行测试程序
./test_logger
```

### 3. 测试流程

1. **基本功能测试** - 程序自动执行，无需交互
2. **等待用户确认** - 程序会提示：
   ```
   === Stress Test ===
   Press Enter to start stress test, or Ctrl+C to exit...
   ```
3. **执行压力测试** - 按 Enter 键开始压力测试
4. **查看测试结果** - 每个测试完成后会显示统计信息

### 4. 中断测试

- 在任何时候按 `Ctrl+C` 可以中断测试并退出程序

```
[Stress Test 1] Sending 10000 log messages rapidly...
  Sent 1000 messages...
  Sent 2000 messages...
  ...
```

### 性能统计

每个测试完成后会显示性能统计：

```
  ✓ Completed: 10000 messages in 1234 ms
  ✓ Throughput: 8103.73 messages/second
```

**统计信息说明：**
- **Completed**: 完成的消息数量和总耗时（毫秒）
- **Throughput**: 吞吐量（消息/秒），用于评估系统性能

### 等待提示

测试之间会有等待时间，确保消息被处理：

```
Waiting 3 seconds for logger to process...
```

## 常见问题

### Q1: 测试过程中程序卡住不动？

**可能原因：**
- Logger 进程未启动或已崩溃
- 消息队列已满
- 数据库写入阻塞

**解决方法：**
1. 检查 Logger 进程是否运行：`ps aux | grep tcu_logger`
2. 检查消息队列状态：`ipcs -q`
3. 查看 Logger 进程日志
4. 重启 Logger 进程

### Q2: 吞吐量低于预期？

**可能原因：**
- 系统负载过高
- 数据库 I/O 性能瓶颈
- 消息队列处理速度慢

**解决方法：**
1. 检查系统负载：`top` 或 `htop`
2. 检查磁盘 I/O：`iostat -x 1`
3. 调整缓冲区配置（需修改代码宏定义）
4. 优化数据库配置

### Q3: 测试数据未完全入库？

**可能原因：**
- 测试结束后立即查询，数据还在缓冲区
- Logger 进程处理速度慢
- 数据库写入失败

**解决方法：**
1. 等待更长时间（建议等待 10-30 秒）
2. 检查 Logger 进程日志是否有错误
3. 检查数据库文件权限
4. 手动触发刷新：`kill -USR1 $(pgrep tcu_logger)`（如果支持）

### Q4: 多线程测试出现数据错误？

**可能原因：**
- 线程同步问题
- 消息队列竞争

**解决方法：**
1. 检查代码中的线程同步机制
2. 验证消息队列的线程安全性
3. 查看是否有重复或丢失的消息

## 测试建议

### 1. 测试环境准备

- 确保系统资源充足（CPU、内存、磁盘空间）
- 关闭不必要的后台程序
- 确保数据库文件有足够的磁盘空间

### 2. 测试顺序

建议按照以下顺序进行测试：

1. 先运行基本功能测试，确保系统正常
2. 再运行压力测试，逐步增加负载
3. 最后验证数据完整性

### 3. 性能优化

如果发现性能问题，可以考虑：

- 调整缓冲区大小（修改 `LOGGER_MAX_BUFFER_SIZE` 宏）
- 调整刷新间隔（修改 `LOGGER_FLUSH_INTERVAL_SECONDS` 宏）
- 优化数据库配置
- 使用更快的存储设备

## 注意事项

1. **资源消耗**：压力测试会产生大量数据，确保有足够的磁盘空间
2. **测试时间**：完整测试可能需要几分钟时间，请耐心等待
3. **系统影响**：压力测试会占用较多系统资源，可能影响其他程序
4. **数据清理**：测试完成后，建议清理测试数据，避免占用过多空间
