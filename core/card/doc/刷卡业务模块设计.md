# 刷卡业务模块设计

## 1. 文档目的

本文用于支撑重构项目中的“刷卡业务模块”设计与实现，目标不是复刻旧工程的函数组织，而是把旧工程里与刷卡相关的：

- 业务流程
- 状态流转
- 数据流
- 读卡器报文
- 主站交互报文
- UI 交互边界

重新整理成一套适合新模块落地的设计说明。

本文基于旧工程中以下代码路径归纳：

- `zk/protocol/deal_card_esam.c`
- `zk/protocol/ic_card.c`
- `zk/protocol/charge_logic.c`
- `zk/protocol/cmd_interface.c`
- `zk/inc/ic_card.h`
- `zk/inc/billing_control_unit.h`
- `zk/inc/charge_logic.h`
- `zk/inc/deal_card_esam.h`
- `c6flushcad.cpp`
- `e1chargeinfo.cpp`
- `f7checkoutok.cpp`
- `a3about.cpp`

## 2. 当前模块边界

如果当前要实现的是“读卡器模块”，建议边界收敛成：

- 只负责读卡器设备交互
- 只负责报文收发、超时、响应解析、命令编排
- 不负责在线/离线业务判定
- 不负责主站授权
- 不负责充电页面跳转

也就是说，读卡器模块应当是一个通用设备模块，例如：

- `reader_transport`
- `reader_device`
- `card_reader`

它的职责只有三类：

1. 把上层请求转换成读卡器命令帧
2. 把串口响应帧解析成结构化结果
3. 提供稳定的原子操作和基础流程

这里要特别强调：

- 在线模式和离线模式不应该成为读卡器模块内部的两套流程
- 在线/离线的区别属于上层刷卡业务状态机
- 读卡器模块只提供“能读什么、怎么读、结果是什么”

建议不要再让 UI 或业务层直接感知：

- 串口帧结构
- 命令码
- 响应码
- 异或校验
- 收发超时
- 重试策略

## 3. 建议的分层方式

建议重构后按 3 层组织：

1. UI / 充电状态机层
2. 刷卡业务层
3. 读卡器模块层

对应关系如下：

- UI / 充电状态机层
  - 决定何时需要“启动刷卡”
  - 决定何时需要“停充刷卡”
  - 消费刷卡结果并切页
- 刷卡业务层
  - 决定当前是在线还是离线
  - 决定只读卡号，还是继续读余额/写卡
  - 决定是否需要向主站发请求
- 读卡器模块层
  - 打开 RF
  - 激活卡片
  - M1 认证
  - 读块/写块
  - 关闭 RF
  - 蜂鸣器控制

这三层里，当前文档的重点应放在第 3 层。

## 4. 读卡器模块应提供的能力

读卡器模块建议只提供“通用能力”，不要直接暴露“在线启动”“离线启动”这种业务名词。

### 4.1 原子命令

建议至少提供以下原子命令：

1. `open_rf`
2. `close_rf`
3. `active_card`
4. `auth_block`
5. `read_block`
6. `write_block`
7. `drive_buzzer`

### 4.2 基础组合流程

在原子命令之上，建议封装少量通用组合流程：

1. `detect_card`
2. `read_card_identity`
3. `read_card_balance_area`
4. `write_card_balance_area`
5. `finish_with_buzzer`

### 4.3 推荐接口

```c
typedef struct ReaderDevice ReaderDevice;

typedef struct {
    uint16_t code;
    uint8_t  payload[256];
    uint16_t payload_len;
} ReaderResult;

int reader_open_rf(ReaderDevice *dev, ReaderResult *out);
int reader_close_rf(ReaderDevice *dev, ReaderResult *out);
int reader_active_card(ReaderDevice *dev, uint16_t timeout_ms, ReaderResult *out);
int reader_auth_block(ReaderDevice *dev, uint8_t key_type, uint8_t block_no,
                      const uint8_t key[6], const uint8_t uid[4], ReaderResult *out);
int reader_read_block(ReaderDevice *dev, uint8_t block_no, ReaderResult *out);
int reader_write_block(ReaderDevice *dev, uint8_t block_no, const uint8_t data[16], ReaderResult *out);
int reader_drive_buzzer(ReaderDevice *dev, uint16_t on_ms, uint8_t repeat, ReaderResult *out);
```

### 4.4 推荐输出对象

```c
typedef struct {
    uint8_t uid[4];
    uint8_t card_no[16];
} ReaderCardIdentity;

typedef struct {
    uint8_t locked_flag;
    uint32_t balance;
    uint32_t balance_neg;
} ReaderCardBalanceInfo;
```

## 5. 上层如何使用读卡器模块

### 5.1 离线刷卡启动

离线启动不需要读卡器模块知道“离线”这个概念。上层只是在读卡后继续多做几步本地校验。

核心业务步骤：

1. 打开读卡器 RF
2. 激活 M1 卡
3. 读取 UID
4. 认证扇区 0
5. 读取块 1，得到卡号
6. 校验是否为预约卡
7. 校验是否在白名单
8. 认证并读取余额/锁状态所在扇区
9. 校验锁卡标志
10. 校验余额及余额反码
11. 余额足够则授权通过
12. 关闭 RF
13. 蜂鸣提示
14. 向充电状态机返回“授权成功”

其中前 5 步都属于通用读卡能力，6 到 10 步属于上层业务层。

### 5.2 在线刷卡启动

在线启动也不需要读卡器模块知道“在线”这个概念。

核心业务步骤：

1. 打开 RF
2. 激活卡片
3. 读取 UID
4. 认证扇区 0
5. 读取块 1，得到卡号
6. 校验是否为预约卡
7. 关闭 RF
8. 蜂鸣提示
9. 置位“请求主站启动授权”
10. 向主站发送启动授权请求
11. 等待主站回包
12. 主站允许则授权成功
13. 主站拒绝则授权失败

其中前 5 步仍然是读卡器模块的通用能力，6 到 13 步属于上层业务层。

### 5.3 离线刷卡停充/结算

这条链路本质是“用启动卡结束本次充电，并在本地完成金额处理”。

核心业务步骤：

1. 用户在充电中刷卡停充
2. 进入停充结算流程
3. 再次寻卡并读取卡号
4. 校验是否为启动卡
5. 读取当前余额/锁状态/结算数据块
6. 计算本次扣费后的新余额
7. 写回卡片相应数据块
8. 关闭 RF
9. 蜂鸣提示
10. 返回结算成功和最终卡余额

其中 3、5、7、8、9 依赖读卡器模块，卡匹配、扣费计算和结果归并属于上层业务层。

### 5.4 在线刷卡停充/结算

在线模式下，刷卡停充更多是“识别到正确卡后，请主站做结算确认”。

核心业务步骤：

1. 用户刷卡请求停充
2. 本地读取卡号
3. 校验是否为启动卡
4. 关闭 RF
5. 蜂鸣提示
6. 通知主站发起停充/结算
7. 等待主站返回结算结果
8. 主站返回余额、账单结果、停机原因
9. 页面跳转到结算完成页

可以看到，在线和离线的区别根本不在读卡器命令序列本身，而在“读卡结果交给谁做决策”。

## 6. 建议的新模块输入输出

### 6.1 输入事件

对于读卡器模块，输入不要设计成业务事件，而应该设计成设备动作请求：

- `OpenRf`
- `CloseRf`
- `ActiveCard`
- `AuthBlock`
- `ReadBlock`
- `WriteBlock`
- `DriveBuzzer`
- `CancelCurrentOp`
- `ResetDevice`

### 6.2 输出结果

对于读卡器模块，输出也应该是设备结果，而不是启动授权结果：

- `ReaderOpSuccess`
- `ReaderOpFailed`
- `ReaderTimeout`
- `ReaderNoCard`
- `ReaderFrameError`
- `ReaderDeviceError`

### 6.3 对 UI 的输出字段

读卡器模块本身不应该直接对 UI 暴露字段。

UI 应消费上层刷卡业务层提供的会话视图，而不是读卡器模块视图。

如果一定要定义读卡器模块的输出结构，建议限制在如下范围：

```c
typedef struct {
    uint16_t op_code;
    uint16_t result_code;
    uint16_t reader_status;
    uint8_t  data[256];
    uint16_t data_len;
} ReaderOpResult;
```

## 7. 读卡器模块的状态机

读卡器模块建议只有设备态和命令执行态，不要引入在线/离线业务态。

### 7.1 顶层设备状态

```text
Idle
Sending
WaitingResponse
Completed
Failed
Resetting
```

### 7.2 推荐的命令执行状态

```text
OpenRf
ActiveCard
AuthBlock
ReadBlock
WriteBlock
CloseRf
DriveBuzzer
```

### 7.3 推荐迁移原则

旧工程里设备操作经常通过：

- `wait_result_pos`
- `wait_result_type`
- `cmd_result`
- `timestart`

推进。

重构时建议改成：

- 单一当前命令
- 单一等待响应状态
- 单一超时器
- 单一结果对象

不要继续保留“多个 flag + 多个等待结果类型”这种写法。

## 8. 数据流总览

### 8.1 读卡器模块通用数据流

```text
BusinessLayer 发起 ReaderOp
  -> ReaderModule 组帧
  -> SerialDriver 发送串口数据
  -> ReaderModule 等待响应
  -> ReaderModule 校验 STX/LEN/XOR/ETX
  -> ReaderModule 解析 status_code 和 payload
  -> ReaderModule 返回 ReaderOpResult
  -> BusinessLayer 决定下一步
```

### 8.2 在线/离线的差异点

```text
离线:
  上层在 read_card_identity 之后继续发起 read_balance / write_block

在线:
  上层在 read_card_identity 之后停止本地读卡流程，转主站授权
```

也就是说，在线/离线的转换点通常就是：

- 已拿到 `uid`
- 已拿到 `card_no`

从这里开始，后续流程由业务层决定。

## 9. 读卡器通信帧定义

旧工程中读卡器串口帧格式是固定的，可直接作为新模块底层协议定义。

### 9.1 请求帧格式

```text
+----------+----------+----------+----------------------+----------+----------+
| STX      | LEN_H    | LEN_L    | PAYLOAD              | XOR      | ETX      |
+----------+----------+----------+----------------------+----------+----------+
| 0x02     | 1 byte   | 1 byte   | LEN bytes            | 1 byte   | 0x03     |
+----------+----------+----------+----------------------+----------+----------+
```

其中：

- `LEN` 表示 `PAYLOAD` 长度
- `PAYLOAD` 前 2 字节是命令码
- `XOR` 是对 `PAYLOAD` 全部字节异或

### 9.2 响应帧格式

旧工程接收处理逻辑表明，响应帧也遵循同一包结构：

- `buf[0] = 0x02`
- `buf[1..2] = payload_len`
- `buf[3..4] = status_code`
- `buf[5..] = result_data`
- 末尾为 `xor + 0x03`

响应解析规则：

- `status_code == 0x0000` 表示命令执行成功
- `status_code != 0x0000` 表示读卡器错误码

新模块建议统一解析为：

```c
typedef struct {
    uint16_t status_code;
    uint8_t  data[256];
    uint16_t data_len;
} ReaderResponse;
```

### 9.3 命令码定义

旧工程已确认的读卡器命令码如下：

| 命令 | 代码 |
| --- | --- |
| 蜂鸣器控制 | `0x3113` |
| 打开 RF | `0x3190` |
| 关闭 RF | `0x3191` |
| 激活卡 | `0x3224` |
| M1 认证 | `0xC202` |
| 读块 | `0xC203` |
| 写块 | `0xC204` |

### 9.4 命令入参定义

#### 打开 RF

- 无参数

#### 关闭 RF

- 无参数

#### 激活卡

- 参数长度：2
- 含义：激活超时时间

旧工程里常见写法是：

```text
00 00
```

即使用默认激活时间；宏上又定义了：

- `ACTIVE_CARD_DELAYTIME = 30000`

说明旧版本曾支持把超时下发给读卡器，但当前分支多处实际下发的是 `00 00`。

#### M1 认证

- 参数长度：12
- 字段布局：

| 偏移 | 长度 | 含义 |
| --- | --- | --- |
| 0 | 1 | Key 类型，旧工程使用 `0x60`，即 KeyA |
| 1 | 1 | 块号 |
| 2 | 6 | 密钥 |
| 8 | 4 | UID |

典型参数：

- 扇区 0 认证：块号 `0x01`
- 扇区 4 认证：块号 `0x10` 或关联块
- 扇区 8 认证：块号 `0x20` 或关联块

#### 读块

- 参数长度：1
- 参数：块号

#### 写块

- 参数长度：17
- 字段布局：

| 偏移 | 长度 | 含义 |
| --- | --- | --- |
| 0 | 1 | 块号 |
| 1 | 16 | 要写入的数据 |

#### 蜂鸣器控制

- 参数长度：3
- 字段布局：

| 偏移 | 长度 | 含义 |
| --- | --- | --- |
| 0 | 2 | 单次响时长 |
| 2 | 1 | 响的次数 |

旧工程典型值：

- `DRIVE_BUZZER_DELAYTIME = 500`
- 次数 `0x01`

### 9.5 关键响应类型

旧工程把成功步骤映射为统一业务响应码，新模块可以直接沿用语义：

| 响应语义 | 代码 |
| --- | --- |
| 蜂鸣完成 | `0x0009` |
| RF 已打开 | `0x000A` |
| RF 已关闭 | `0x000B` |
| 卡已激活 | `0x000C` |
| 扇区 0 认证完成 | `0x000D` |
| 块 1 已读取 | `0x000E` |
| 扇区 4 读认证完成 | `0x000F` |
| 扇区 4 写认证完成 | `0x0010` |
| 块 16 已读取 | `0x0011` |
| 块 16 已写入 | `0x0012` |
| 扇区 8 读认证完成 | `0x0013` |
| 扇区 8 写认证完成 | `0x0014` |
| 块 32 已读取 | `0x0015` |
| 块 32 已写入 | `0x0016` |
| 读卡器命令失败 | `0xFFFF` |

### 9.6 关键错误码

底层读卡器相关错误码：

| 错误语义 | 代码 |
| --- | --- |
| 蜂鸣失败 | `0x7001` |
| 打开 RF 失败 | `0x7002` |
| 关闭 RF 失败 | `0x7003` |
| 激活卡失败 | `0x7004` |
| M1 认证失败 | `0x7005` |
| 读块失败 | `0x7006` |
| 写块失败 | `0x7007` |
| 读卡器响应超时 | `0x7008` |

旧工程中还有一个特殊读卡器错误：

- `0x3006`

它在“等待刷卡但当前还没刷到卡”时被当作可重试结果处理，而不是立即失败。

新模块建议显式定义为：

- `READER_NO_CARD_RETRYABLE`

## 10. 主站交互数据流与帧内容

### 10.1 在线启动授权请求

旧工程在刷到卡并完成本地识别后，会构造一帧“请求主站启动授权”的数据，长度固定为 42 字节。

字段布局如下：

| 偏移 | 长度 | 含义 | 当前观察值 |
| --- | --- | --- | --- |
| 0 | 1 | 消息类型 | `0x01` |
| 1 | 1 | 业务类型/原因码 | `0x03` |
| 2 | 1 | 保留 | `0x00` |
| 3 | 1 | 地址扩展 | `0x00` |
| 4 | 1 | 支付/渠道类型 | `0x00` |
| 5 | 3 | 保留 | `0x00 0x00 0x00` |
| 8 | 1 | 启动方式 | `0x02`，表示刷卡启动 |
| 9 | 16 | 卡号，旧工程做了 `+0x30` 处理 | 16 字节 |
| 25 | 1 | 附加信息类型 | `0x04` |
| 26..41 | 16 | 保留/扩展 | `0x00...` |

注意事项：

- 旧工程对 `card_no[16]` 做的是“每字节加 `0x30`”处理，不是标准 BCD 到 ASCII 的双字节展开
- 这意味着旧协议中的卡号编码方式不够自描述
- 新模块建议在内部统一保存为原始字节数组，在 HostAdapter 层单独做协议转换

建议新模块内部 DTO：

```c
typedef struct {
    uint8_t  port;
    uint8_t  start_mode;      // card_online
    uint8_t  card_uid[4];
    uint8_t  card_no[16];
    uint8_t  is_bespeak;
    uint8_t  bespeak_card_no[16];
} HostStartAuthRequest;
```

### 10.2 在线启动授权响应

旧工程最终关注的不是原始函数调用，而是以下几个结果位：

- `host_start_charge_flag_1 = YES`
- `host_refuse_card_start_charge = YES`

以及主站可能附带回来的：

- 卡余额
- 流水号
- 订单信息

建议新模块抽象成：

```c
typedef struct {
    uint8_t  accepted;
    uint16_t reason;
    uint32_t card_balance;
    uint8_t  order_no[32];
    uint8_t  extra[64];
    uint16_t extra_len;
} HostStartAuthResponse;
```

### 10.3 在线停充/结算请求

旧工程里在线停充也是通过标志位触发，由主站接口层去发送。对新模块而言，更合适的抽象是：

```c
typedef struct {
    uint8_t  port;
    uint8_t  card_no[16];
    uint32_t start_money;
    uint32_t current_money;
    uint32_t charge_energy;
    uint32_t charge_duration;
} HostStopBillingRequest;
```

### 10.4 在线停充/结算响应

建议统一抽象成：

```c
typedef struct {
    uint8_t  accepted;
    uint16_t reason;
    uint32_t card_balance;
    uint32_t end_money;
    uint8_t  stop_reason;
    uint8_t  extra[64];
    uint16_t extra_len;
} HostStopBillingResponse;
```

## 11. 业务数据对象定义建议

### 11.1 卡片身份信息

```c
typedef struct {
    uint8_t uid[4];
    uint8_t card_no[16];
    uint8_t locked_flag;
    uint32_t balance;
} CardIdentity;
```

### 11.2 上层刷卡会话

这个对象不属于读卡器模块本身，而属于刷卡业务层。

```c
typedef enum {
    CARD_SCENE_NONE = 0,
    CARD_SCENE_START,
    CARD_SCENE_STOP,
    CARD_SCENE_QUERY,
} CardScene;

typedef enum {
    CARD_MODE_OFFLINE = 0,
    CARD_MODE_ONLINE,
} CardMode;

typedef enum {
    CARD_PHASE_IDLE = 0,
    CARD_PHASE_OPEN_RF,
    CARD_PHASE_WAIT_CARD,
    CARD_PHASE_AUTH_S0,
    CARD_PHASE_READ_BLOCK1,
    CARD_PHASE_READ_EXTRA,
    CARD_PHASE_WRITE_EXTRA,
    CARD_PHASE_CLOSE_RF,
    CARD_PHASE_BUZZER,
    CARD_PHASE_WAIT_HOST,
    CARD_PHASE_DONE,
    CARD_PHASE_FAILED,
} CardPhase;

typedef struct {
    uint8_t port;
    CardScene scene;
    CardMode mode;
    CardPhase phase;
    CardIdentity card;
    uint8_t start_card_no[16];
    uint8_t bespeak_card_no[16];
    uint8_t card_found;
    uint8_t need_whitelist_check;
    uint8_t need_bespeak_check;
    uint8_t need_lock_check;
    uint8_t need_balance_check;
    uint32_t start_money;
    uint32_t end_money;
    uint16_t error_code;
    uint32_t deadline_ms;
} CardSession;
```

## 12. 在线/离线状态转换建议

这里不再把它们定义成读卡器模块状态，而是定义成上层业务层状态转换。

### 12.1 启动场景

```text
Idle
  -> WaitCard
  -> ReadIdentity
  -> DecideMode
     -> OfflineCheckLocal
     -> OnlineWaitHost
  -> Authorized / Rejected
```

### 12.2 停充场景

```text
Charging
  -> WaitStopCard
  -> ReadIdentity
  -> VerifyStartCard
     -> OfflineBilling
     -> OnlineWaitBilling
  -> Finished / Rejected
```

### 12.3 关键结论

在线/离线的差异应该只出现在：

- `ReadIdentity` 之后
- `VerifyStartCard` 之后

在这两个节点之前，读卡器模块行为完全可以复用。

## 13. 推荐的错误码分层

建议不要再把所有错误都混在一起，而是分层定义：

### 13.1 读卡器错误

- 打开 RF 失败
- 关闭 RF 失败
- 激活卡失败
- 认证失败
- 读块失败
- 写块失败
- 响应超时
- 未检测到卡

### 13.2 卡业务错误

- 非预约卡
- 非白名单卡
- 非启动卡
- 卡锁定
- 卡余额校验失败
- 卡余额不足

### 13.3 主站错误

- 主站拒绝启动
- 主站拒绝停充
- 主站授权超时
- 主站结算超时

### 13.4 会话错误

- 用户取消
- 枪已断开
- 充电状态不允许刷卡
- 会话已失效

## 14. UI 与刷卡模块的边界

### 14.1 C6 页面

只负责：

- 让用户选择“刷卡启动”
- 告诉刷卡模块进入 `start + card`
- 展示“请刷卡 / 正在识别 / 授权成功 / 授权失败”

不负责：

- 轮询底层命令队列
- 直接解析卡号
- 直接判断余额、预约卡、白名单

### 14.2 E1 页面

只负责：

- 让用户触发“刷卡停充”
- 展示“请刷启动卡结束充电”
- 接收停充/结算结果

### 14.3 F7 页面

只负责展示：

- 结束金额
- 卡余额
- 停机原因
- 结算成功或失败

## 16. 需要重点保留的旧行为

重构时建议保留以下行为，因为它们已经构成旧系统的业务约束：

1. 在线启动必须区分“本地识卡成功”和“主站授权成功”两个阶段
2. 未刷到卡时，读卡器 `0x3006` 应视为可重试
3. 离线启动要支持预约卡校验
4. 离线启动要支持白名单校验
5. 离线启动和离线结算都要校验余额与余额反码
6. 停充时要校验“停止卡是否等于启动卡”
7. 刷卡成功后要有蜂鸣反馈
8. 枪已断开时必须中止刷卡会话

## 17. 重构落地建议

既然当前聚焦的是读卡器模块，建议按下面顺序实现：

### 第一步：实现串口帧收发

先实现：

- 组帧
- 收帧
- 超时
- 命令/响应映射
- STX/LEN/XOR/ETX 校验

先不接 UI，不接主站，不接在线/离线业务。

### 第二步：实现通用原子命令接口

先做：

- 打开 RF
- 关闭 RF
- 激活卡
- 认证块
- 读块
- 写块
- 蜂鸣器控制

### 第三步：实现通用组合流程

增加：

- `read_card_identity`
- `read_card_balance_area`
- `write_card_balance_area`

### 第四步：上层再接在线/离线状态机

这一步不属于读卡器模块本身，由刷卡业务层完成：

- 离线本地校验
- 在线主站授权
- 离线本地结算
- 在线主站结算

## 18. 当前结论

对当前阶段而言，旧工程真正需要继承的不是原函数名，也不是在线/离线两套读卡流程，而是下面这 5 件事：

1. 读卡器协议帧格式
2. 通用读卡动作序列
3. 原子命令和响应码映射
4. 未刷到卡时的可重试策略
5. 在线/离线差异只放在上层业务状态转换

如果按这个边界重构，新模块可以同时满足：

- 易测试
- 易替换底层读卡器
- 易适配不同主站协议
- UI 不再和刷卡底层耦合
