# 读卡器协议栈设计

## 1. 范围

本文只聚焦读卡器协议栈，不讨论在线/离线刷卡业务本身。

本文的目标是把旧工程中的读卡器协议收敛成重构时可直接实现的“设备协议模块”说明。

## 2. 协议栈职责

`ic_card.c` 在旧工程里的真实职责很明确：

1. 从上层命令队列中取一条命令
2. 按固定帧格式组帧
3. 通过串口发送给读卡器
4. 等待读卡器响应
5. 校验响应帧合法性
6. 把响应映射成统一的结果码
7. 回填到命令队列，供上层业务继续推进

它不负责：

- 启动充电还是停充结算
- 在线授权还是离线校验
- UI 页面切换
- 余额业务规则

所以重构时应该把它视为：

- `ReaderTransport`
- 或 `CardReaderProtocol`

而不是“刷卡业务模块”本身。

## 3. 旧工程中的协议栈架构

旧工程的交互模型本质是“单设备、单通道、命令队列驱动”。

### 3.1 输入

输入来自：

- `cmd_info.cmd_BCU_to_cardreader`

上层把要执行的读卡命令写入这个队列，包括：

- `cmd_type`
- `cmd_para_len`
- `cmd_in_buf`

### 3.2 输出

输出仍回写到同一个队列项：

- `cmd_result`
- `cmd_result_len`
- `cmd_re_buf`
- `cmd_errno`

### 3.3 执行模型

协议栈不是并发执行多条命令，而是：

- 用 `cmd_in_pos` 表示写入位置
- 用 `cmd_out_pos` 表示当前待执行位置
- 每次只处理一条命令
- 一条命令完成后再推进 `cmd_out_pos`

所以这实际上是一个串行的、单飞行中的命令协议层。

## 4. 协议栈状态机

### 4.1 端口状态

`ic_card.h` 中定义了 3 个核心状态：

| 状态 | 值 | 含义 |
| --- | --- | --- |
| `IC_CARD_PORT_IDLE` | `0` | 空闲，可取下一条命令 |
| `IC_CARD_PORT_TRANSING_TO_IED` | `1` | 正在发送给读卡器 |
| `IC_CARD_PORT_WAIT_IED_REPLY` | `2` | 已发送完成，等待响应 |

这三个状态已经足够构成重构版本的底层状态机。

### 4.2 关键运行字段

`IC_CARD_PORT_FLAG` 中最关键的协议态字段有：

| 字段 | 含义 |
| --- | --- |
| `ic_card_mon_state` | 当前端口状态 |
| `ic_card_mon_need_reply` | 当前命令是否期待响应 |
| `ic_card_poll_type` | 当前正在等待哪种协议命令的响应 |
| `ic_card_begin_poll_ied_time` | 本轮轮询开始时间 |
| `ic_card_begin_wait_ied_time` | 开始等待响应的时间 |
| `ic_card_recv_dl` | 接收缓冲区当前已跳过的偏移 |

这里的核心设计思想是：

- 用 `ic_card_poll_type` 记住“当前发出去的是哪种底层命令”
- 收到响应后，再结合命令队列里的 `cmd_type` 映射成上层结果码

### 4.3 状态转换

协议栈层面的状态转换非常简单：

```text
IDLE
  -> 取出一条待执行命令
  -> 组帧并发送
  -> TRANSING_TO_IED
  -> 发送完成
  -> WAIT_IED_REPLY
  -> 收到完整合法响应
  -> 解析并回填结果
  -> IDLE

WAIT_IED_REPLY
  -> 超时
  -> 回填错误
  -> IDLE
```

这个模型不需要关心在线/离线业务。

## 5. 命令调度模型

### 5.1 调度入口

核心调度在 `Ic_Card_Scheduler()`。

其行为是：

1. 仅在 `IDLE` 态尝试取下一条命令
2. 取 `cmd_out_pos` 指向的队列项
3. 如果该命令被取消，则直接跳过
4. 否则按 `cmd_type` 选择不同组帧函数

### 5.2 命令到组帧函数的映射

| `cmd_type` | 底层组帧函数 |
| --- | --- |
| `DRIVE_BUZZER` | `control_card_reader_buzz()` |
| `OPEN_RF` | `open_card_reader_RF()` |
| `CLOSE_RF` | `close_card_reader_RF()` |
| `ACTIVE_CARD` | `active_card()` |
| `CERTIFY_M1_CARD_*` | `certify_m1_card()` |
| `READ_BLOCK*_DATA` | `read_block_data()` |
| `WRITE_BLOCK*_DATA` | `write_block_data()` |

这说明旧工程已经把协议层压缩成了 7 类底层原子动作。

### 5.3 重构建议

重构时建议保留这个抽象，不要继续暴露更多细碎函数名，而是统一成：

```c
typedef enum {
    READER_OP_BUZZER,
    READER_OP_OPEN_RF,
    READER_OP_CLOSE_RF,
    READER_OP_ACTIVE_CARD,
    READER_OP_AUTH_M1,
    READER_OP_READ_BLOCK,
    READER_OP_WRITE_BLOCK,
} ReaderOpCode;
```

## 6. 请求帧格式

`ic_card.c` 里的所有发包都遵循同一格式：

```text
+----------+----------+----------+----------------------+----------+----------+
| STX      | LEN_H    | LEN_L    | PAYLOAD              | XOR      | ETX      |
+----------+----------+----------+----------------------+----------+----------+
| 0x02     | 1 byte   | 1 byte   | LEN bytes            | 1 byte   | 0x03     |
+----------+----------+----------+----------------------+----------+----------+
```

其中：

- `STX = 0x02`
- `LEN = PAYLOAD` 长度
- `PAYLOAD` 前 2 字节固定是命令码
- `XOR = PAYLOAD` 全部字节异或
- `ETX = 0x03`

### 6.1 示例：打开 RF

```text
02 00 02 31 90 A1 03
```

解释：

- `00 02` 表示 payload 长度为 2
- `31 90` 表示 `CODE_OPEN_RF`
- `A1 = 0x31 ^ 0x90`

### 6.2 示例：关闭 RF

```text
02 00 02 31 91 A0 03
```

### 6.3 示例：激活卡

```text
02 00 04 32 24 XX XX XOR 03
```

其中 `XX XX` 是激活延时参数。

### 6.4 示例：M1 认证

```text
02 00 0E C2 02 [12-byte param] XOR 03
```

12 字节参数布局：

| 偏移 | 长度 | 含义 |
| --- | --- | --- |
| 0 | 1 | key type，旧工程通常是 `0x60` |
| 1 | 1 | block no |
| 2 | 6 | key |
| 8 | 4 | uid |

### 6.5 示例：读块

```text
02 00 03 C2 03 [block] XOR 03
```

### 6.6 示例：写块

```text
02 00 13 C2 04 [block + 16-byte data] XOR 03
```

### 6.7 示例：蜂鸣器

```text
02 00 05 31 13 [time_h time_l repeat] XOR 03
```

## 7. 响应帧格式

接收处理逻辑在 `ic_card_recv()` 和 `accept_higher_order_deal_recv()`。

从代码可还原出的响应格式为：

```text
+----------+----------+----------+----------------------+----------+----------+
| STX      | LEN_H    | LEN_L    | STATUS + DATA        | XOR      | ETX      |
+----------+----------+----------+----------------------+----------+----------+
| 0x02     | 1 byte   | 1 byte   | LEN bytes            | 1 byte   | 0x03     |
+----------+----------+----------+----------------------+----------+----------+
```

其中：

- `STATUS = 2 bytes`
- `DATA = LEN - 2 bytes`

响应解析规则：

1. 从接收缓冲区中找到 `0x02`
2. 读取长度 `LEN`
3. 判断缓冲区是否已经收到完整一帧
4. 对 `STATUS + DATA` 做 XOR 校验
5. 校验尾字节是否为 `0x03`
6. 取 `STATUS`
7. `STATUS == 0x0000` 视为协议命令执行成功
8. `STATUS != 0x0000` 视为读卡器错误码

### 7.1 成功响应映射

成功时，协议栈不会直接把 `0x0000` 交给上层，而是映射为业务语义结果码。

例如：

- `CODE_OPEN_RF` -> `RF_OPENED`
- `CODE_CLOSE_RF` -> `RF_CLOSED`
- `CODE_ACTIVE_CARD` -> `CARD_ACTIVED`

对于认证/读块/写块，还会二次结合命令队列中的 `cmd_type` 做细分：

- `CERTIFY_M1_CARD_S0` -> `M1_CARD_S0_CERTIFIED`
- `CERTIFY_M1_CARD_S4_R` -> `M1_CARD_S4_CERTIFIED_R`
- `CERTIFY_M1_CARD_S4_W` -> `M1_CARD_S4_CERTIFIED_W`
- `READ_BLOCK1_DATA` -> `BLOCK1_DATA_READED`
- `READ_BLOCK16_DATA` -> `BLOCK16_DATA_READED`
- `WRITE_BLOCK16_DATA` -> `BLOCK16_DATA_WRITED`

这说明：

- `ic_card_poll_type` 只知道底层大类命令
- `cmd_type` 负责恢复更具体的业务语义

### 7.2 失败响应映射

失败时：

- `cmd_result = CARDR_TO_BCU_CMD_RESULT_ERROR`
- `cmd_errno = reader_status_code`

也就是说，失败场景直接把读卡器原始状态码上抛了。

## 8. 协议层超时与异常处理

### 8.1 超时策略

`ic_card.h` 中定义：

- `IC_CARD_RECV_OUT_TIME_VALUE = 2000`

等待响应超过 2 秒时，协议层直接判超时。

超时处理结果：

- `cmd_result = CARDR_TO_BCU_CMD_RESULT_ERROR`
- `cmd_errno = READER_RESPONSE_OVERTIME`
- 端口状态回到 `IDLE`

### 8.2 帧级异常

`ic_card_recv()` 中实际做了三类帧检查：

1. 包头检查
2. XOR 校验检查
3. 包尾 `0x03` 检查

如果失败，旧代码大多直接继续等待直到超时，然后归并为超时错误。

这也是旧协议栈的一个问题：

- 帧格式错误
- 校验错误
- 接收不完整

最终都容易被归并成“等待超时”。

### 8.3 重构建议

重构时建议明确拆分为：

- `READER_ERR_TIMEOUT`
- `READER_ERR_BAD_STX`
- `READER_ERR_BAD_LENGTH`
- `READER_ERR_BAD_XOR`
- `READER_ERR_BAD_ETX`
- `READER_ERR_DEVICE_STATUS`

这样后续联调会容易很多。

## 10. 推荐的重构模型

### 10.1 模块拆分

建议拆成两层：

1. `reader_transport`
2. `reader_protocol`

职责区分：

- `reader_transport`
  - 串口发送
  - 串口接收缓存
  - 原始字节收发
- `reader_protocol`
  - 组帧
  - 解帧
  - 校验
  - 命令队列
  - 超时
  - 结果映射

### 10.2 核心数据结构建议

```c
typedef enum {
    READER_STATE_IDLE = 0,
    READER_STATE_SENDING,
    READER_STATE_WAITING_REPLY,
} ReaderState;

typedef enum {
    READER_OP_NONE = 0,
    READER_OP_BUZZER,
    READER_OP_OPEN_RF,
    READER_OP_CLOSE_RF,
    READER_OP_ACTIVE_CARD,
    READER_OP_AUTH_M1,
    READER_OP_READ_BLOCK,
    READER_OP_WRITE_BLOCK,
} ReaderOpCode;

typedef struct {
    ReaderOpCode op;
    uint8_t params[32];
    uint8_t param_len;
    uint8_t canceled;
} ReaderCommand;

typedef struct {
    uint16_t result_code;
    uint16_t device_status;
    uint8_t data[256];
    uint16_t data_len;
} ReaderReply;

typedef struct {
    ReaderState state;
    ReaderOpCode waiting_op;
    uint32_t tx_start_ms;
    uint32_t rx_start_ms;
    uint16_t rx_offset;
} ReaderProtocolContext;
```

### 10.3 对上层暴露的最小接口

```c
int reader_submit(const ReaderCommand *cmd);
int reader_poll(void);
int reader_try_fetch_result(ReaderReply *reply);
void reader_reset(void);
```

## 11. 对当前重构的直接结论

如果你当前只做读卡器模块，那么真正要实现的是下面这 6 件事：

1. 命令队列
2. 三态协议状态机
3. 固定帧格式组帧
4. 收包校验与解帧
5. 底层命令到结果码的映射
6. 超时与错误归并

而不需要在这个模块里直接实现：

- 在线刷卡流程
- 离线刷卡流程
- 余额业务规则
- 主站授权流程
- 页面跳转流程

在线/离线只会影响“上层提交给读卡器模块的命令序列”，不会改变读卡器协议栈本身。
