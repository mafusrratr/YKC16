# 充电计费单元软件概要设计评审稿

## 1. 项目目标与范围
- 本项目目标是构建一套面向充电桩控制器的可扩展软件架构，覆盖桩控、计费、电表采集、平台通信四大能力。
- 当前评审范围：`pile_controller`、`tcu_logic`、`tcu_meter`、`tcu_comm` 及其 MQTT 协作接口。
- 不在本次范围：平台全部业务帧细节闭环、全部异常场景压测与生产化加固。

## 2. 总体架构设计
- 架构形态：多进程微服务 + MQTT 核心路由总线。
- 进程职责拆分：
  - `tcu_pile_controller`：设备协议与桩侧交互（CAN/多接入扩展）。
  - `tcu_logic`：状态机、业务规则、计费策略、交易记录生成。
  - `tcu_meter`：RS485/645 电表采集。
  - `tcu_comm`：平台协议适配（按平台独立实现）。
  - `logger/daemon`：日志持久化与进程守护。
- 设计原则：核心业务稳定、协议适配外置、接口契约化、模块可替换。

## 3. 模块间接口与消息契约
- 统一消息总线前缀：`tcu/...`
- 核心主题：
  - `tcu/pile/{gun}/cmd|event|data`
  - `tcu/logic/{gun}/cmd|event|feeData`
  - `tcu/meter/{gun}/data|event`
  - `tcu/plat/{gun}/cmd|event`
- QoS 策略：
  - 命令/关键事件：高可靠（QoS2/QoS1）
  - 遥测/高频数据：低延迟（QoS0）
- 统一字段：`ts/seq/source/gun`，保证可追踪、可回放、可对账。

## 4. 关键业务流程设计
- 启动流程：上层命令 -> `tcu_logic` 鉴权与状态校验 -> 下发 `pile cmd` -> 接收 `start_response/start_complete` -> 进入 `CHARGING`。
- 充电中：`tcu_meter` 上送电量电压电流，`tcu_logic` 输出 `feeData` 与状态事件，`tcu_comm` 组帧上送平台。
- 停机流程：命令/故障触发 -> `STOPPING` 收敛 -> `STOPPED` -> 生成 `update_record` -> 平台确认 -> 日志确认闭环。

## 5. 状态机与控制策略
- 业务状态：`IDLE / PREPARE / STARTING / CHARGING / STOPPING / STOPPED / ERROR`。
- 核心策略：
  - 启动只在合法状态响应；
  - 启停重发窗口与超时控制；
  - 停机收敛窗口（防止电量结算丢失）；
  - 故障事件触发停机与恢复路径分离。
- 状态机作为单一业务真源，平台和设备层只做输入输出适配。

## 6. 计费与记录设计
- 按时段模型计费：支持 `timeSeg/chargeFee/serviceFee`。
- `feeData` 输出总量与分段累计，单位和量纲统一。
- 交易记录包含：订单标识、起止时间、SOC、电量、费用分解、时段明细。
- 平台确认后执行本地记录确认，形成闭环。

## 7. 平台适配设计
- 平台适配独立目录，互不污染：
  - `core/comm/ChangzhouLiyangSmartcityPlat`
  - `core/comm/Zhongshihua2.0`
- 统一内部语义：平台帧 -> `tcu/plat/{gun}/cmd|event` -> `tcu_logic`。
- 进展：主登录链路、心跳、计费模型、实时监测、启动/停机/记录关键帧已具备基础实现，后续按帧持续补齐。

## 8. 架构优势（本方案价值）
- 解耦：协议变化不侵入核心业务。
- 隔离：单模块故障不拖垮全系统。
- 并行：多团队按 Topic 契约并行开发。
- 可观测：消息天然可抓包、可追踪、可回放。
- 可扩展：新增平台/新设备接入成本低。
- 可演进：适合长期维护与版本迭代。
