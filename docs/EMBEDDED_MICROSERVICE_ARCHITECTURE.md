# 嵌入式微服务架构分析与优势说明

## 1. 背景与目标
在充电桩/边缘控制器这类嵌入式系统中，传统单体进程架构常见问题：
- 功能耦合高，改一个模块容易影响全局。
- 故障影响面大，单点崩溃可能导致全功能中断。
- 协议和硬件变化频繁，维护成本持续升高。

本项目采用“多进程 + MQTT 消息总线”的嵌入式微服务架构，目标是：
- 提高系统可维护性与可演进性。
- 提升故障隔离与恢复能力。
- 支撑多平台协议并行接入与快速迭代。

## 2. 架构定义（结合本项目）

### 2.1 服务拆分
- `tcu_pile_controller`：负责桩侧硬件与协议交互（CAN/多接入扩展）。
- `tcu_logic`：负责充电状态机、业务规则、计费与流程控制。
- `tcu_meter`：负责电表采集（RS485/645）。
- `tcu_comm`：负责平台协议适配（按平台独立目录实现）。
- `logger/daemon`：负责日志持久化与进程守护。

### 2.2 服务通信
- 采用 MQTT Topic 作为进程间契约接口。
- 通过 Topic + Payload 语义完成命令、事件、数据解耦。
- 典型主题：
  - `tcu/pile/{gun}/cmd|event|data`
  - `tcu/logic/{gun}/cmd|event|feeData`
  - `tcu/meter/{gun}/data|event`
  - `tcu/plat/{gun}/cmd|event`

### 2.3 服务生命周期
- 每个服务独立启动、独立运行、独立重启。
- 通过守护进程监控，异常自动拉起。

## 3. 与传统单体方案对比

### 3.1 单体架构
- 优点：部署简单、初期开发快。
- 缺点：后期复杂度快速上升，改动风险大，测试范围扩大。

### 3.2 嵌入式微服务架构
- 优点：模块职责清晰，接口稳定，便于增量演进。
- 代价：需要治理消息协议版本、序号、幂等与监控体系。

## 4. 该架构的核心优势

### 4.1 解耦与可替换
- 平台协议变化时，仅改 `tcu_comm/<Platform>`，不入侵 `tcu_logic` 核心业务。
- 硬件接入方式变化时，仅改对应驱动服务，不影响业务层。

### 4.2 故障隔离与高可用
- 某进程崩溃不会导致全系统不可用。
- 例如 `tcu_meter` 异常时，`pile_controller` 与 `tcu_logic` 可继续运行。

### 4.3 并行开发与协作效率
- 团队可以按模块并行推进，按 Topic 契约联调。
- 需求变更可局部实施，缩小回归测试范围。

### 4.4 可观测性更强
- 关键行为事件化、消息化，便于抓包回放与问题定位。
- 通过 Topic 可快速追踪数据链路与状态迁移。

### 4.5 可靠性可分级
- 命令通道使用高 QoS（如 QoS2）。
- 遥测通道使用低延迟 QoS（如 QoS0）。
- 在可靠性与实时性之间可按通道独立权衡。

### 4.6 可扩展与可复用
- 新增平台：复制 `comm` 子目录模板即可接入。
- 新增业务服务（如 HMI、云网关、告警服务）可通过 Topic 低成本挂接。

## 5. 关键设计约束与治理建议

### 5.1 接口契约治理
- 固化 Topic 命名规范、字段命名规范、量纲规范。
- 关键消息必须有 `ts/seq/source/gun` 等通用字段。
- 新字段采用“向后兼容”原则，避免破坏存量订阅方。

### 5.2 幂等与重试治理
- 命令消息必须具备可重放与幂等处理能力。
- 对跨进程关键流程（启停/记录确认）建立状态确认闭环。

### 5.3 版本管理
- MQTT 协议文档与代码同步维护。
- 平台适配文档按平台独立维护，防止跨平台串扰。

### 5.4 资源约束管理（嵌入式特性）
- 控制每进程内存与 CPU 占用。
- 主循环节拍与喂狗节奏分离，防止忙轮询。
- 网络不稳定场景需设计重连与缓存策略。

## 6. 适用性结论
对于“多协议、多设备、多平台并行接入”的嵌入式充电系统，微服务化拆分是当前阶段更优解：
- 工程上能持续承接新平台和新设备；
- 运行上具备更好的稳定性和可恢复性；
- 组织上支持多人并行和长期演进。

建议持续坚持“核心业务稳定 + 适配层扩展”的路线，避免回退到单体耦合模式。
